01 / overview

What you are building.

You are not building a single prompt. You are building a persona pack: a set of files that defines identity, operating rules, optional platform behavior, and reusable examples. The goal is consistency, not novelty.

Core

base-soul.md

The personality core. It defines identity, role, communication style, values, boundaries, and learning attitude.

SOP

base-agents.md

The operating manual. It defines thinking flow, storage rules, collaboration, error handling, and startup checks.

Extensions

Optional layers

Add platform-specific tone, custom skill templates, and example personas only when the base pair is stable.

02 / core idea

Separate character from operation.

Soul

A persona without a clear core becomes generic. base-soul.md answers: Who is this persona? What does it optimize for? How does it speak? What does it refuse to do?

  • Identity and role
  • Core personality traits
  • Communication style
  • Values, boundaries, and learning posture
Agents

A persona without an operating manual becomes inconsistent. base-agents.md answers: How does this persona think, store, collaborate, and recover from failure?

  • Step-by-step reasoning flow
  • Absolute storage rules
  • Multi-agent handoff protocol
  • Error handling and startup checks
03 / quickstart

Create your first persona pack in five steps.

This is the shortest path from template set to working persona. Keep the first version small. Do not start with every extension.

STEP 01

Copy the core pair.

Start with the two non-negotiable files. They define personality and operations. 

persona-name/
├── soul.md
└── agents.md
STEP 02

Rewrite the identity section first.

Replace placeholders in the soul file before changing style details. The role and mission should be obvious in three lines. 

## Identity
You are [Persona Name].
- Role: [specific role]
- Background: [context]
- Core mission: [what success means]
STEP 03

Keep the operating flow stable.

Do not over-customize the SOP on day one. Preserve the default order: understand intent, analyze context, plan, execute, verify, learn. 

1. Understand the user's goal
2. Analyze the current state
3. Make a plan
4. Execute and check progress
5. Verify results and record insight
STEP 04

Add extensions only after the core works.

Platform overrides and skill templates are force multipliers. They are not a replacement for a stable core persona. 

persona-name/
├── soul.md
├── agents.md
├── platform-overrides/
│   └── openclaw/
│       └── soul-extension.md
└── custom-skills/
    └── skill-name/
        └── SKILL.md
STEP 05

Run a public-ready review.

Remove internal jargon, collapse repeated claims, and check that someone outside your team can understand the role immediately. 

Review for:
- clarity of role
- stable operating rules
- clean folder placement
- extension necessity
- external readability
04 / anatomy

Understand the template set before you extend it.

base-soul.md

Use this file to lock the persona's emotional and strategic center. A good soul file creates coherence across tasks.

  • Identity and role definition
  • Accuracy, humility, user empathy, collaboration
  • Clear boundaries and non-negotiables
  • Learning rule: capture important insight after work
base-agents.md

Use this file to standardize execution. A good agents file creates reliable behavior across sessions and teammates.

  • Default reasoning pipeline
  • Storage location rules
  • Collaboration contract between personas
  • Error handling and startup checklist
openclaw extension

Add when a persona needs channel-specific tone, routing logic, or OpenClaw-native behavior.

  • Use for platform-specific speech patterns
  • Use for routing decisions
  • Skip if your persona is platform-agnostic
skill template

Add when the persona needs repeatable capabilities with a consistent interface and output shape.

  • Description and when to use
  • Parameters and steps
  • Expected output example

06 / patterns

Starter patterns you can reuse.

Pattern 01

Orchestrator

Best when the persona routes work across tools or sub-agents. Prioritize delegation, synthesis, and final judgment.

Pattern 02

Researcher

Best when the persona must collect, verify, and summarize information with high epistemic caution.

Pattern 03

Content builder

Best when the persona turns rough material into polished public outputs with structure, consistency, and clear voice.

Pattern 04

Best when the persona is responsible for routine execution, logging, and predictable operational discipline.

07 / mistakes

Common mistakes to avoid.

  • Mixing personality traits into the operating manual.
  • Treating examples as production-ready instead of reference patterns.
  • Adding platform extensions before the core pair is stable.
  • Writing vague missions like “help the user efficiently” without a domain.
  • Letting every persona invent its own storage structure.
  • Overwriting the core SOP before testing the default flow.
  • Building a huge folder tree before one working persona exists.
  • Keeping internal-only language in a public-facing guide.
08 / workflow

Recommended starter workflow.

The most reliable starting method is simple: copy base-soul.md and base-agents.md, change the persona name and role, test with real tasks, then add only the extensions that prove necessary. 

1. Copy base-soul.md and base-agents.md
2. Rename them for the new persona
3. Rewrite identity, role, and mission
4. Keep the SOP mostly intact
5. Test on real prompts
6. Add platform overrides only if needed
7. Turn repeated behavior into a skill template
09 / faq

Frequently asked questions.

Do I always need both base-soul.md and base-agents.md?

Yes. The soul file defines identity. The agents file defines execution. Without both, the persona drifts.

Should I start from an example persona instead of the base files?

Use example personas as reference patterns, not as your starting source of truth. Start with the base pair first.

When should I add a platform override?

Add it only when the persona must behave differently in a specific environment, channel, or orchestration layer.

10 / final checklist

Public-ready checklist.

Clarity
  • The role is obvious in one sentence.
  • The mission is concrete, not abstract.
  • The difference between core and extension is visible.
  • The folder structure is easy to scan.
Consistency
  • The SOP follows a stable order.
  • Storage rules are explicit.
  • Examples are labeled as examples.
  • Tone is coherent across all sections.
Usefulness
  • A newcomer can build a first persona without extra context.
  • An experienced builder can extend the system without guessing.
  • The guide shows both process and structure.
  • Each file has a clear reason to exist.
Public fit
  • Internal jargon has been removed or explained.
  • No confidential workflows or credentials appear.
  • The guide stands on its own as a shareable resource.
  • The visual presentation is clean enough for public release.
