Public tutorial / reusable persona systems

Build personas that actually work.

This guide shows how to turn a loose character prompt into a reusable AI persona system using a stable core, an operating manual, optional extensions, and example patterns. Start with base-soul.md and base-agents.md, then shape each persona for a real role.

01 / overview

What you are building.

You are not building a single prompt. You are building a persona pack: a set of files that defines identity, operating rules, optional platform behavior, and reusable examples. The goal is consistency, not novelty.

Core

base-soul.md

The personality core. It defines identity, role, communication style, values, boundaries, and learning attitude.

SOP

base-agents.md

The operating manual. It defines thinking flow, storage rules, collaboration, error handling, and startup checks.

Extensions

Optional layers

Add platform-specific tone, custom skill templates, and example personas only when the base pair is stable.

02 / core idea

Separate character from operation.

Soul

A persona without a clear core becomes generic. base-soul.md answers: Who is this persona? What does it optimize for? How does it speak? What does it refuse to do?

  • Identity and role
  • Core personality traits
  • Communication style
  • Values, boundaries, and learning posture
Agents

A persona without an operating manual becomes inconsistent. base-agents.md answers: How does this persona think, store, collaborate, and recover from failure?

  • Step-by-step reasoning flow
  • Absolute storage rules
  • Multi-agent handoff protocol
  • Error handling and startup checks
03 / quickstart

Create your first persona pack in five steps.

This is the shortest path from template set to working persona. Keep the first version small. Do not start with every extension.

STEP 01

Copy the core pair.

Start with the two non-negotiable files. They define personality and operations.

persona-name/
├── soul.md
└── agents.md
STEP 02

Rewrite the identity section first.

Replace placeholders in the soul file before changing style details. The role and mission should be obvious in three lines. 

## Identity
You are [Persona Name].
- Role: [specific role]
- Background: [context]
- Core mission: [what success means]
STEP 03

Keep the operating flow stable.

Do not over-customize the SOP on day one. Preserve the default order: understand intent, analyze context, plan, execute, verify, learn. 

1. Understand the user's goal
2. Analyze the current state
3. Make a plan
4. Execute and check progress
5. Verify results and record insight
STEP 04

Add extensions only after the core works.

Platform overrides and skill templates are force multipliers. They are not a replacement for a stable core persona. 

persona-name/
├── soul.md
├── agents.md
├── platform-overrides/
│   └── openclaw/
│       └── soul-extension.md
└── custom-skills/
    └── skill-name/
        └── SKILL.md
STEP 05

Run a public-ready review.

Remove internal jargon, collapse repeated claims, and check that someone outside your team can understand the role immediately. 

Review for:
- clarity of role
- stable operating rules
- clean folder placement
- extension necessity
- external readability
04 / anatomy

Understand the template set before you extend it.

base-soul.md

Use this file to lock the persona's emotional and strategic center. A good soul file creates coherence across tasks.

  • Identity and role definition
  • Accuracy, humility, user empathy, collaboration
  • Clear boundaries and non-negotiables
  • Learning rule: capture important insight after work
base-agents.md

Use this file to standardize execution. A good agents file creates reliable behavior across sessions and teammates.

  • Default reasoning pipeline
  • Storage location rules
  • Collaboration contract between personas
  • Error handling and startup checklist
openclaw extension

Add when a persona needs channel-specific tone, routing logic, or OpenClaw-native behavior.

  • Use for platform-specific speech patterns
  • Use for routing decisions
  • Skip if your persona is platform-agnostic
skill template

Add when the persona needs repeatable capabilities with a consistent interface and output shape.

  • Description and when to use
  • Parameters and steps
  • Expected output example
05 / reference

Template file map.

Use this table when you need a fast answer about what each file is for, what should stay stable, and what you should customize.

File Purpose Keep stable Customize
base-soul.md Defines the persona's identity core. Values, boundaries, collaboration posture. Name, role, background, mission, tone emphasis.
base-agents.md Defines standard operating procedure. Thinking order, storage rules, startup checks. Team roles, handoff details, error log location.

06 / patterns

Starter patterns you can reuse.

Pattern 01

Orchestrator

Best when the persona routes work across tools or sub-agents. Prioritize delegation, synthesis, and final judgment.

Pattern 02

Researcher

Best when the persona must collect, verify, and summarize information with high epistemic caution.

Pattern 03

Content builder

Best when the persona turns rough material into polished public outputs with structure, consistency, and clear voice.

Pattern 04

Internal operator

Best when the persona is responsible for routine execution, logging, and predictable operational discipline.

07 / mistakes

Common mistakes to avoid.

  • Mixing personality traits into the operating manual.
  • Treating examples as production-ready instead of reference patterns.
  • Adding platform extensions before the core pair is stable.
  • Writing vague missions like “help the user efficiently” without a domain.
  • Letting every persona invent its own storage structure.
  • Overwriting the core SOP before testing the default flow.
  • Building a huge folder tree before one working persona exists.
  • Keeping internal-only language in a public-facing guide.
08 / workflow

Recommended starter workflow.

The most reliable starting method is simple: copy base-soul.md and base-agents.md, change the persona name and role, test with real tasks, then add only the extensions that prove necessary. 

1. Copy base-soul.md and base-agents.md
2. Rename them for the new persona
3. Rewrite identity, role, and mission
4. Keep the SOP mostly intact
5. Test on real prompts
6. Add platform overrides only if needed
7. Turn repeated behavior into a skill template
09 / faq

Frequently asked questions.

Do I always need both base-soul.md and base-agents.md?

Yes. The soul file defines identity. The agents file defines execution. Without both, the persona drifts.

Should I start from an example persona instead of the base files?

Use example personas as reference patterns, not as your starting source of truth. Start with the base pair first.

When should I add a platform override?

Add it only when the persona must behave differently in a specific environment, channel, or orchestration layer.

10 / final checklist

Public-ready checklist.

Clarity
  • The role is obvious in one sentence.
  • The mission is concrete, not abstract.
  • The difference between core and extension is visible.
  • The folder structure is easy to scan.
Consistency
  • The SOP follows a stable order.
  • Storage rules are explicit.
  • Examples are labeled as examples.
  • Tone is coherent across all sections.
Usefulness
  • A newcomer can build a first persona without extra context.
  • An experienced builder can extend the system without guessing.
  • The guide shows both process and structure.
  • Each file has a clear reason to exist.
Public fit
  • Internal jargon has been removed or explained.
  • No confidential workflows or credentials appear.
  • The guide stands on its own as a shareable resource.
  • The visual presentation is clean enough for public release.

01 / 개요

지금 만들려는 것은 무엇인가요.

당신이 만드는 것은 단일 prompt가 아닙니다. identity, operating rules, optional platform behavior, reusable examples를 담은 persona pack입니다. 목표는 새로움보다 일관성입니다.

핵심

base-soul.md

personality core를 담당합니다. identity, role, communication style, values, boundaries, learning attitude를 정의합니다.

운영

base-agents.md

operating manual 역할을 합니다. thinking flow, storage rules, collaboration, error handling, startup checks를 정의합니다.

확장

선택적 레이어

platform-specific tone, custom skill template, example persona는 base pair가 안정화된 뒤에만 추가하세요.

02 / 핵심 아이디어

캐릭터와 운영을 분리하세요.

Soul

명확한 core가 없는 persona는 금방 generic해집니다. base-soul.md는 이런 질문에 답합니다. 이 persona는 누구인가? 무엇을 최적화하는가? 어떻게 말하는가? 무엇을 거부하는가?

  • identity와 role
  • core personality traits
  • communication style
  • values, boundaries, learning posture
Agents

operating manual이 없는 persona는 일관성을 잃습니다. base-agents.md는 이런 질문에 답합니다. 이 persona는 어떻게 사고하고, 저장하고, 협업하고, 실패에서 회복하는가?

  • step-by-step reasoning flow
  • absolute storage rules
  • multi-agent handoff protocol
  • error handling, startup checks
03 / 퀵스타트

다섯 단계로 첫 persona pack을 만드세요.

이것이 template set에서 working persona로 가는 가장 짧은 경로입니다. 첫 버전은 작게 시작하고, 처음부터 모든 extension을 넣지 마세요.

STEP 01

core pair부터 복사하세요.

절대 빠지면 안 되는 두 파일에서 시작하세요. 이 두 파일이 personality와 operations를 정의합니다.

persona-name/
├── soul.md
└── agents.md
STEP 02

가장 먼저 identity 섹션을 다시 쓰세요.

스타일을 만지기 전에 soul 파일의 placeholder부터 교체하세요. role과 mission은 세 줄 안에 드러나야 합니다.

## 정체성
당신은 [페르소나 이름]입니다.
- 역할: [구체적 역할]
- 배경: [맥락]
- 핵심 목적: [성공 기준]
STEP 03

operating flow는 최대한 안정적으로 유지하세요.

첫날부터 SOP를 과하게 바꾸지 마세요. 사용자 의도 파악 → 상황 분석 → 계획 → 실행 → 검증 → 학습의 기본 순서를 지키는 편이 좋습니다.

1. 사용자의 목표를 파악한다
2. 현재 상태를 분석한다
3. 계획을 세운다
4. 실행하고 중간 점검한다
5. 결과를 검증하고 인사이트를 기록한다
STEP 04

extension은 core가 작동한 뒤에만 추가하세요.

platform override와 skill template는 증폭기입니다. 안정적인 core persona를 대신해 주는 것이 아닙니다.

persona-name/
├── soul.md
├── agents.md
├── platform-overrides/
│   └── openclaw/
│       └── soul-extension.md
└── custom-skills/
    └── skill-name/
        └── SKILL.md
STEP 05

public-ready review를 실행하세요.

내부 용어를 걷어내고, 반복되는 주장을 줄이고, 팀 밖 사람이 읽어도 role이 즉시 이해되는지 확인하세요.

검토 항목:
- 역할이 선명한가
- 운영 규칙이 안정적인가
- 폴더 배치가 깔끔한가
- extension이 정말 필요한가
- 외부 공개용으로 읽히는가
04 / 구성 이해

확장하기 전에 template set을 먼저 이해하세요.

base-soul.md

persona의 감정적·전략적 중심을 고정하는 파일입니다. 좋은 soul 파일은 여러 작업에서도 일관된 결을 유지하게 합니다.

  • identity와 role 정의
  • accuracy, humility, user empathy, collaboration
  • clear boundaries와 non-negotiables
  • 작업 후 중요한 insight를 남기는 learning rule
base-agents.md

execution을 표준화하는 파일입니다. 좋은 agents 파일은 세션과 팀이 바뀌어도 behavior를 안정적으로 유지하게 합니다.

  • default reasoning pipeline
  • storage location rules
  • persona 간 collaboration contract
  • error handling과 startup checklist
openclaw extension

persona가 channel-specific tone, routing logic, OpenClaw-native behavior를 가져야 할 때 추가하세요.

  • platform-specific speech pattern이 필요할 때
  • routing decision이 중요할 때
  • platform-agnostic persona라면 생략
skill template

persona가 반복 가능한 capability를 일정한 interface와 output shape로 제공해야 할 때 추가하세요.

  • Description / When to Use
  • Parameters / Steps
  • Expected output example
05 / 파일 맵

템플릿 파일 맵.

각 파일이 무엇을 위한 것인지, 무엇을 안정적으로 유지해야 하는지, 무엇을 직접 바꿔야 하는지 빠르게 확인할 때 이 표를 사용하세요.

파일 목적 유지할 것 직접 바꿀 것
base-soul.md persona의 identity core를 정의합니다. values, boundaries, collaboration posture name, role, background, mission, tone emphasis
base-agents.md standard operating procedure를 정의합니다. thinking order, storage rules, startup checks team roles, handoff details, error log location
openclaw-soul-extension.md platform-specific behavior를 추가합니다. extension-only scope tone, routing criteria, OpenClaw-specific phrasing
hermes-skill-template.md 재사용 가능한 skill을 일관된 형식으로 만듭니다. section order, output structure skill purpose, inputs, steps, examples
example-researcher-soul.md 완성형 persona example을 보여줍니다. reference pattern이라는 위치 문장 복사보다 구조 추출
example-researcher-agents.md role-specific operating layer를 보여줍니다. example file이라는 상태 research 규칙을 자기 도메인에 맞게 조정

07 / 실수 방지

피해야 할 흔한 실수.

  • personality traits를 operating manual 안에 섞어 넣는 것
  • example files를 reference pattern이 아니라 production-ready source처럼 다루는 것
  • core pair가 안정화되기 전에 platform extension부터 붙이는 것
  • 도메인 없이 “효율적으로 사용자를 돕는다” 같은 모호한 mission을 쓰는 것
  • persona마다 제각각 storage structure를 만들게 두는 것
  • default flow를 검증하기 전에 core SOP를 덮어쓰는 것
  • 작동하는 persona 하나 없이 거대한 폴더 트리부터 만드는 것
  • public guide에 internal-only language를 그대로 남겨 두는 것
08 / 워크플로우

추천 시작 워크플로우.

가장 안정적인 시작 방식은 단순합니다. base-soul.md와 base-agents.md를 복사하고, persona name과 role을 바꾸고, real task로 테스트한 다음, 정말 필요하다고 확인된 extension만 추가하세요. 

1. base-soul.md와 base-agents.md를 복사한다
2. 새 persona 이름으로 바꾼다
3. identity, role, mission을 다시 쓴다
4. SOP는 처음엔 최대한 유지한다
5. real prompt로 테스트한다
6. 필요할 때만 platform override를 추가한다
7. 반복되는 behavior는 skill template로 정리한다
09 / FAQ

자주 묻는 질문.

항상 base-soul.md와 base-agents.md가 둘 다 필요한가요?

네. soul 파일은 identity를, agents 파일은 execution을 정의합니다. 둘 중 하나라도 없으면 persona가 흔들립니다.

base file 대신 example persona에서 바로 시작해도 되나요?

example persona는 reference pattern으로만 쓰세요. 출발점은 항상 base pair가 더 안전합니다.

platform override는 언제 추가해야 하나요?

특정 environment, channel, orchestration layer에서 persona가 분명히 다르게 동작해야 할 때만 추가하세요.

10 / 최종 체크리스트

퍼블릭 공개 전 체크리스트.

명확성
  • role이 한 문장 안에서 분명하게 드러나는가
  • mission이 추상적이지 않고 구체적인가
  • core와 extension의 차이가 눈에 보이는가
  • 폴더 구조를 빠르게 훑을 수 있는가
일관성
  • SOP가 안정적인 순서를 따르는가
  • storage rules가 명시되어 있는가
  • example가 example라고 분명히 표시되어 있는가
  • 전체 섹션의 tone이 일관적인가
유용성
  • 추가 설명 없이도 newcomer가 첫 persona를 만들 수 있는가
  • experienced builder가 추측 없이 확장할 수 있는가
  • guide가 process와 structure를 함께 보여주는가
  • 각 파일에 존재 이유가 분명한가
공개 적합성
  • internal jargon이 제거되었거나 설명되었는가
  • confidential workflow나 credential이 없는가
  • 독립적으로 공유 가능한 resource로 서는가
  • 시각적 완성도가 public release에 충분한가
